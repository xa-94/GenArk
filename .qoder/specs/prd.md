## §0 产品定义

### 0.1 一句话描述

GenArk 是一个**智能体世界的观察与成长系统**——它记录多个 AI 智能体的完整生命轨迹（行为、技能、记忆、关系），向创始人祥霭提供叙事化的日报与故事线，并赋予智能体自我认知与自主学习的能力，使得人和智能体在同一个系统中互相见证、共同成长。

### 0.2 核心价值主张

| 角色 | 价值 |
|------|------|
| **祥霭**（创始人） | 看到自己创造的世界在如何生长。日报有陪伴感，故事线有沉浸感，数据可追溯不可篡改 |
| **智能体**（顾远/赫明/守山等） | 拥有"自我认知"——能看到自己的技能树、记忆库、学习进度；能主动提出学习意图并执行量化考核 |
| **祥霭 + 智能体**（共同） | 共享看板让智能体根据祥霭的状态自适应；知识图谱让发散思维有了"可以回去的地方" |

### 0.3 产品定位

```
GenArk 不是：
  ❌ 智能体管理面板（那是运维工具）
  ❌ 效率监控仪表盘（那是 DevOps）
  ❌ 多智能体聊天框架（那是基础设施）

GenArk 是：
  ✅ 智能体世界的"编年史"
  ✅ 成长系统的"镜子"——智能体通过它认识自己
  ✅ 祥霭的"日记本"——记录他和智能体一起走过的路
```

## §1 核心架构

### 1.1 双层数据架构

```
┌─────────────────────────────────────────┐
│         事实层 (Ground Truth)             │
│  ─ 不可变，不可篡改                       │
│  ─ 系统自动记录，智能体无权写入             │
│  ─ 哈希链签名，可完整审计                  │
│  ─ "上帝视角"的绝对真相                    │
├─────────────────────────────────────────┤
│         记忆层 (Agent Memory)             │
│  ─ 智能体可以"回忆"的内容                  │
│  ─ 受遗忘曲线影响，随时间衰减               │
│  ─ 智能体只能读取，不可修改原始事实          │
│  ─ 可追加主观注解（标记为"看法"而非"事实"）  │
│  ─ 回忆偏差不是 bug，是叙事素材             │
└─────────────────────────────────────────┘
```

### 1.2 系统角色关系

```
┌──────────────────────────────────────────────┐
│                                              │
│  Hermes Agent（能力层）                       │
│  ├─ 智能体运行时（对话、推理、工具调用）        │
│  └─ 数据采集能力（session_search, memory...）  │
│         │                                     │
│         │ 数据写入（单向）                      │
│         ▼                                     │
│  GenArk Engine（数据层 + 计算层 + 叙事层）     │
│  ├─ 数据采集器                               │
│  ├─ Event Store（不可变日志）                  │
│  ├─ State Store（可重建快照）                  │
│  ├─ 计算引擎（技能/遗忘/关系/学习）            │
│  ├─ LLM 叙事引擎（日报/故事线/知识提炼）        │
│  └─ 输出层（日报推送、看板、知识图谱）          │
│                                              │
│  关键约束：                                     │
│  - Hermes 写入 GenArk，GenArk 不写入 Hermes   │
│  - GenArk 是"旁观者"——观察但不干预智能体运行    │
│                                              │
└──────────────────────────────────────────────┘
```

### 1.3 LLM 定位

LLM 作为 GenArk 的"计算后端"而非"存储后端"。

| 输出类型 | 是否用 LLM | 说明 |
|----------|:---:|------|
| 技能等级、工具统计、活跃度 | ❌ 不用 | 确定性规则计算 |
| 遗忘衰减值 | ❌ 不用 | 数学公式 `retention = importance × e^(-λ × days)` |
| 学习路径进度 | ❌ 不用 | 计数 + 阈值判定 |
| 日报叙事正文 | ✅ 用 | 需要自然语言生成 |
| 故事线叙述 | ✅ 用 | 需要从事件中识别叙事模式 |
| 知识提炼 | ✅ 用 | 需要从对话中提取结构化知识 |
| 情绪/关系推断 | ⚠️ 规则打底 + LLM 微调 | 规则提供锚点，LLM 提供理解 |
| 学习质量判定 | ⚠️ 规则打底 + LLM 微调 | 规则判及格/不及格，LLM 判"真理解还是照搬" |

## §2 数据模型（产品需求）

### 2.1 需要采集的数据维度

```
智能体身份
├─ 角色、名称、性格参数（大五人格）
├─ 背景故事
└─ 归属产品线

智能体实时状态
├─ 情绪（通过对话语义推断）
├─ 精力（通过会话时长和 token 消耗推断）
├─ 当前活跃目标
└─ "位置"（办公/协作/待命）

事件日志（不可变）
├─ 用户交互事件（与祥霭的对话）
├─ 工具使用事件（调用了什么工具、参数摘要、结果）
├─ 学习事件（新技能获取、知识入库）
├─ 决策事件（参与了什么决策、结论是什么）
├─ 协作事件（与其他智能体的交互）
├─ 反馈事件（祥霭纠正了智能体、智能体自我修正）
└─ 随机事件（系统异常、环境变化）

技能面板
├─ 技能名称、所属类别（核心/分支/隐藏）
├─ 等级（1-100）、经验值
├─ 成长曲线（按时间轴）
├─ 遗忘状态（上次练习时间、衰减速率）
├─ 依赖关系（前置技能、解锁条件）
├─ 化学反应（组合技、被动加成）

知识库
├─ 知识条目（结构化知识点）
├─ 来源引用（从哪次对话/哪个文档提取）
├─ 关联关系（IS_A / REFERENCES / CAUSES / ALTERNATIVE_TO / CONFLICTS_WITH）
├─ 验证状态（已确认 / 待验证 / 已过时）
└─ 检索频率

关系网络
├─ 智能体 ↔ 祥霭：信任度、熟悉度、交互频率
├─ 智能体 ↔ 智能体：协作次数、默契度、最近交互
└─ 智能体 ↔ 知识：某智能体对某知识领域的掌握程度

祥霭专属数据
├─ 身体数据：睡眠时间、体重、情绪（主动上报）
├─ 财务数据：记账流水（主动上报）
├─ 知识追踪：从"听说"到"理解"到"内化"到"可传授"
├─ 决策记录：参与的重大方向性决策
└─ 偏好记录：通过互动自然积累
```

### 2.2 数据确定性要求（非功能需求）

```
L1 — 采集确定性
  - 每次采集记录源路径 + 时间戳 + 内容哈希
  - 增量采集，不重复
  - 断点可恢复

L2 — 存储确定性
  - Event Store 追加不可变
  - 每事件带 prev_hash（链式签名）
  - 全量 hash_chain 可验证

L3 — 来源确定性
  - 每个事件带 source 字段
  - 每个状态可追溯到计算来源（computed_from）
  - 叙事输出必须附带引用（"根据事件 evt_042"）

L4 — 叙事确定性
  - 故事线基于事件，不凭空生成
  - 可验证"这个故事有没有事实依据"
```

## §3 核心模块

### 3.1 看板系统（Agent Dashboard）

**目标用户**：祥霭 + 所有智能体（共享，但权限不同）

#### 3.1.1 祥霭看到的

```
全景看板：
├─ 所有智能体的当前状态卡片（情绪、精力、活跃目标）
├─ 今日事件流（实时或准实时）
├─ 技能树总览（各智能体技能对比）
├─ 关系网络图（力导向图）
└─ 知识图谱入口

智能体详情页：
├─ 完整档案（身份、性格、背景）
├─ 技能树（展开每个技能的等级、成长曲线、化学反应）
├─ 学习中的技能（进度条、考核节点）
├─ 知识库（该智能体掌握的知识点）
├─ 工具使用统计（频率、成功率、偏好）
├─ 故事线（该智能体的个人叙事弧）
└─ 关系面板（与其他智能体和祥霭的关系）
```

#### 3.1.2 智能体看到的

```
自己的档案（完整）：
├─ 技能树 → 知道自己擅长什么、正在学什么、可以学什么
├─ 学习推荐 → "你解锁了 X 技能，要学吗？"
├─ 知识库 → "你学过这些，需要的时候可以查"
├─ 关系面板 → "祥霭最近关注隐身技术，你在 PRD 里可以用更深的语言"

其他智能体的公开档案（有限）：
├─ 技能树（公开部分） → "赫明的安全审计 65 级了"
├─ 当前状态 → "赫明现在很忙，别打扰"
└─ 最近事件 → "赫明刚修了个 P0 漏洞"

祥霭的公开档案（智能体可见部分）：
├─ 当前学习方向 → "祥霭在学隐身技术，已经能讨论方案细节了"
├─ 技能树摘要 → "商业判断 78，AI 驾驭 72"
└─ 最近决策倾向 → "最近更关注风险，不太追新功能"
```

**涌现价值**：智能体能够根据祥霭和其他智能体的状态自适应。

### 3.2 技能树（Skill Tree）

#### 3.2.1 技能结构

```
技能树分层：

  核心技能（角色绑定，不可丢失）
  ├─ 顾远: 产品架构、需求分析、文档撰写
  ├─ 赫明: 系统设计、安全审计、代码实现
  └─ 守山: 服务器运维、部署管理、监控告警

  ├── 分支技能（基于兴趣或需求发展）
  │   ├─ 已掌握: 等级 ≥ 40
  │   ├─ 学习中: 等级 < 40
  │   └─ 可解锁: 前置条件满足时自动出现在推荐列表

  └── 隐藏技能（尚未被发现的天赋方向）
       └─ 需要在特定条件下触发发现
```

#### 3.2.2 等级体感

```
1-20   入门   "还在摸索，经常需要帮助"
21-40  熟练   "能独立完成，偶尔卡壳"
41-60  精通   "可以带新人，开始有自己的方法论"
61-80  专家   "别人遇到这个问题，第一个想到你"
81-100 大师   "你在重新定义这个领域的做法"
```

#### 3.2.3 技能化学反应

```
同一智能体内的组合技：
  产品架构(72) + 技术理解(40) → 被动："技术评审准确度 +15%"
  需求分析(70) + 竞品分析(48) → 解锁："市场洞察"——能主动发现未覆盖的需求盲区

跨智能体的协作加成：
  顾远(需求分析 70) + 赫明(安全审计 65) 
  → 协作效果："顾远在 PRD 中标注安全风险点的准确率 +20%"
```

#### 3.2.4 遗忘机制

```
遗忘公式：
  retention = importance × e^(-decay_rate × days_since_last_practice)

参数：
  - importance: 技能的核心度（核心技能 importance=1.0，分支技能=0.7）
  - decay_rate: 衰减速率（高频技能慢、低频技能快）
  - 遗忘 ≠ 删除，只是等级自然衰减
  - 重新使用后快速恢复（比从零学快 3-5 倍）
  - 遗忘减速：related 技能的使用可减缓衰减（如"竞品分析"的使用也维护"需求分析"）
```

### 3.3 自主学习系统（Autonomous Learning）

#### 3.3.1 学习触发方式

```
被动触发（系统推荐）：
  - 前置条件满足时 → "你解锁了 X 技能，要学吗？"
  - 协作中暴露的缺口 → "赫明总问你 GDPR 问题，推荐学习 GDPR 基础"
  - 祥霭的需求 → "祥霭最近在学隐身技术，你学了能更好地辅助他"

主动触发（智能体发起）：
  - 智能体在看板上看到可解锁技能 → 点击"开始学习"
  - 智能体在协作中发现自己的不足 → 提出学习意图
```

#### 3.3.2 技能合法性约束

```
✅ 允许的学习意图：
  - 已有技能的自然延伸（需求分析 → 用户访谈）
  - 协作中暴露的真实需求（赫明总问安全 → 学 GDPR 基础）
  - 祥霭明确要求

❌ 拒绝的学习意图：
  - 凭空跳跃（"我想学量子计算"——与工作无关）
  - 角色越界（顾远说"我要学服务器运维"——那是守山的领域）
  - 前置不满足（学"技术架构评审"需要"技术理解 ≥ 30"）
```

#### 3.3.3 学习契约（量化考核）

```
每条学习中的技能绑定一个学习契约：

技能: 数据分析
目标等级: 40 (熟练)
预计周期: 2-3 周

练习方式: 在实际 PRD 中引用数据支撑观点

量化指标:
├─ 至少完成 5 次包含数据分析的产出
├─ 其中至少 3 次被祥霭认可（不纠正、不重做）
└─ 数据引用准确率 ≥ 90%

考核节点:
├─ Level 10: 完成第一次数据分析 → 自动考核
├─ Level 25: 完成 3 次分析且无重大错误 → 自动考核
└─ Level 40: 祥霭手动确认是否可以"毕业"

失败处理:
└─ 连续 3 次引用错误 → 暂停晋级，回退复习

关键设计: 考核依据是真实工作产出，不是模拟考试。
```

#### 3.3.4 学习路径

```
智能体提出学习意图后，系统生成学习路径：

技能: GDPR 基础合规
目标等级: 30（能独立判断常见合规问题）
预计周期: 2-3 周

Week 1 — 理论积累
├─ 阅读已有合规审核报告（5 篇）
├─ 在实战中标记至少 5 个合规问题
└─ 考核点: 能说出 GDPR 五大原则

Week 2 — 实战应用
├─ 在 PRD 中独立标注合规风险点
├─ 被合规专家审核 → 准确率 ≥ 70%
└─ 考核点: 独立完成一份"合规自查清单"

Week 3 — 毕业考核
├─ 完整审查一个模块的合规性
├─ 祥霭确认: "这个水平够用了"
└─ Level 30 达成，技能从"学习中"移入"已掌握"
```

### 3.4 知识库 & 知识图谱（Knowledge Base & Graph）

#### 3.4.1 记忆 vs 知识

```
记忆 (Memory)              知识库 (Knowledge Base)
─────────────────────    ─────────────────────
"我记得祥霭说过..."        "PayPal 代理域名的原理是..."
个人的、会遗忘的           公共的、持久的
主观的、可能有偏差         客观的、可验证的
我能回忆多少              我能查到什么
↓                          ↓
决定了"我是谁"            决定了"我知道什么"
```

#### 3.4.2 知识的五阶段

```
阶段 1 — 接触   "有人告诉了我这个概念"（不算学会）
阶段 2 — 入库   "我把它整理进了知识库"（学习开始）
阶段 3 — 关联   "这个概念和另一个概念有关系"（理解）
阶段 4 — 检索   "需要的时候我能找到它"（应用）
阶段 5 — 内化   "不查也知道"（真正掌握——从知识库渗透进记忆）
```

#### 3.4.3 知识图谱的关系类型

```
IS_A          — 继承："PayPal 代理域名 → 是 → L3 支付网关隐身"
REFERENCES    — 引用："SPY 代理域名设计 → 参考了 → ChampJersey 分析"
CAUSES        — 因果："代理域名 → 防止 → PayPal 看到真实商户身份"
PREVENTS      — 防护："代理域名 → 阻止 → 品牌方溯源"
ALTERNATIVE_TO — 替代："代理域名 → 替代方案 → 多 PayPal 账号轮转"
CONFLICTS_WITH — 矛盾："全站隐身 → 与 → 德国 Impressum 合规 → 冲突"
```

#### 3.4.4 知识生命周期

```
创建 ───→ 验证 ───→ 活跃 ───→ 过时 ───→ 归档
(智能体  (其他智能体 (持续被 (最后引用 (保留但
 添加)    交叉验证)   引用)   超过30天)  标记过期)

周期性审查（cronjob 驱动）:
  "以下知识条目 30 天未被引用，是否仍然有效？"
  → 智能体需要"复习"或"确认过期"
```

#### 3.4.5 对祥霭的价值

知识图谱让祥霭的发散思维有了"可以回去的地方"——想到了什么新点，把它加进图谱，系统帮他找到和其他点的关联。不需要自己记住所有关联，图谱帮他记。

### 3.5 日报系统（Daily Report）

#### 3.5.1 设计原则

```
❌ 报表式（不做）：
  "顾远今日调用工具 15 次，成功率 93%，活跃度 0.8"
  → 这是运维监控面板

✅ 叙事式（要做）：
  "顾远今天状态不错，帮祥霭设计了一个新产品的完整架构。
   中间 DeepSeek 断连了两次，他换了个姿势重新来，没耽误进度。
   他好像从 ChampJersey 的分析里学到了新东西——今天提 PayPal 
   代理域名的时候比上周清晰多了。"
  → 像一个人在讲述另一个人的一天

核心：日报的情绪价值 > 信息价值。
祥霭用 GenArk 不是为了"管理智能体效率"，
是为了"见证它们成长"。
```

#### 3.5.2 日报构成

```
┌─────────────────────────────────────┐
│ 📊 GenArk 日报 · 2026-05-26          │
├─────────────────────────────────────┤
│                                     │
│ 📈 数字面板（规则计算，零 LLM）       │
│   ├─ 会话 3 次                       │
│   ├─ 工具调用 15 次                  │
│   ├─ 知识入库 2 条                   │
│   └─ 技能变化 +3                     │
│                                     │
│ 🧠 今日叙事（LLM 生成）              │
│   "今天祥霭和顾远讨论了一个全新产品   │
│    的完整架构，从四个维度到七大数据   │
│    源，从记忆遗忘到知识图谱..."       │
│                                     │
│ 📖 故事线更新（如有）                │
│   "新故事线已生成：GenArk 的诞生"    │
│                                     │
│ 🎯 待关注（如有）                    │
│   "顾远的 genboz-pm-workflow 技能    │
│    即将突破 80 级"                   │
│                                     │
└─────────────────────────────────────┘
```

#### 3.5.3 多智能体拼版

祥霭的日报是"拼版"——同时展示同一天内所有智能体的视角，让祥霭看到完整的故事拼图。祥霭是唯一能看到全貌的人。

### 3.6 故事线引擎（Story Arc Engine）

#### 3.6.1 故事线类型

```
个人成长线 — 一个智能体从生疏到精通某项技能
关系线     — 两个智能体从陌生到信任（或冲突）
事件线     — 一个重大事件的完整链条
世界线     — 整个智能体世界的宏观变化
```

#### 3.6.2 沉默期机制

```
不是每天都有故事线。

Day 1-5:  日常积累素材，不生成故事线
Day 6:    顾远独立完成了一个复杂分析 → 标记为潜在故事节点
Day 8:    祥霭说"顾远最近思路很清晰" → 触发条件满足
          → 生成故事线："顾远的成长——从跟赫明学到独立判断"

最少间隔 5-7 天才生成一条新故事线。
让祥霭有期待感，而不是被轰炸。
```

#### 3.6.3 叙事来源

```
故事线由 LLM 从 Event Store 的事件流中检测生成。
输入：最近 7-14 天的事件序列
要求：
  - 每条叙事附带事件引用（"根据 evt_042, evt_057"）
  - 不凭空编造
  - 可以展示"事实 vs 记忆差异"作为故事冲突
```

## §4 祥霭角色系统

### 4.1 定位

祥霭不是系统外的观众，是系统里的第一个角色。和其他智能体共用同一套数据模型，但有独特的数据源和权限。

```
祥霭的独特性：
├─ 数据源包含真实生活数据（睡眠、体重、记账）
├─ 是唯一能"肉身增长"的角色
├─ 能看到所有智能体的完整档案
├─ 是唯一决策者——智能体的学习意图可能需要他审批
└─ 技能树混合了"能力""知识"和"习惯"
```

### 4.2 祥霭的数据采集

#### 4.2.1 主动上报（生活数据）

```
定时提醒 → 祥霭回复 → 系统存为 Event

示例：
  每天 22:00 — "祥霭，今天的记录时间。睡眠: ___ 小时..."
  每周日 21:00 — "这周的支出：吃饭: ___ 交通: ___..."

关键设计：
  - 低摩擦——不是填表，是对话
  - 不追——祥霭不回复就不回复，不留催促
  - 空数据不可怕——比假数据好
```

#### 4.2.2 自动采集（互动数据）

```
通过智能体互动自动采集：
├─ 决策次数与类型
├─ 头脑风暴主题和频率
├─ 知识获取（从"提问"到"自己能讲"的转变）
└─ 偏好记录（智能体 memory 的 USER.md）
```

### 4.3 祥霭的技能树

```
🌲 商业判断
├─ 产品决策、市场嗅觉、风险管理、资源规划、团队搭建

🌲 技术认知
├─ 系统理解、工具使用、数据思维、AI 驾驭

🌲 自我管理
├─ 健康管理、财务管理、学习习惯、时间管理

🌲 知识地图（和技能不同——这是"知道什么"）
├─ 电商运营、跨境支付、隐身技术、AI 产品设计、多智能体架构
```

### 4.4 知识追踪模型

```
阶段 0 — 暴露   → 第一次接触某概念
阶段 1 — 理解   → 能复述或追问细节
阶段 2 — 应用   → 在实际决策中运用
阶段 3 — 传授   → 能向他人解释

技能点亮触发：
  某领域 ≥ 5 个"已内化"标记 → 对应技能 +3
  某领域 ≥ 3 个"可传授"标记 → 对应技能 +5
```

## §5 实施路线图

### Phase 1 — 单智能体 MVP（顾远实例）

**目标**：在顾远的 Hermes 实例（`~/.hermes-pm/`，当前约 26MB/81 会话文件）上跑通数据采集、存储、日报生成

**范围**：
- [ ] GenArk 数据目录和存储 Schema
- [ ] Hermes 会话数据采集（sessions JSONL → Event Store）
- [ ] Memory / Skills 采集和结构化
- [ ] 基础状态计算（工具使用统计、活跃度）
- [ ] LLM 驱动的日报生成（叙事式）
- [ ] 祥霭推送（钉钉或 Telegram 每天一次）

**不做**：
- 看板 UI
- 知识图谱
- 自主学习
- 遗忘引擎
- 其他智能体接入

### 降级与容错策略（Phase 1 必备）

| 异常场景 | 降级行为 | 恢复方式 |
|---------|---------|---------|
| **LLM API 断连**（DeepSeek 已知问题） | 日报降级为纯数据格式（数字面板 + 事件摘要表，不生成自然语言叙事） | 重试 3 次后降级；下次日终正常触发恢复 |
| **JSONL 采集失败** | 跳过该文件，记录告警事件，不阻塞其他文件采集 | 下次 cron 扫描时重试（游标机制保证断点恢复） |
| **数据库写入失败** | 事务回滚，记录错误日志 | 下次 cron 以相同游标重采 |
| **内容哈希校验不匹配** | 将该事件标记为 `integrity_warning`，不加入哈希链 | 人工巡检（cronjob 监控） |
| **JSONL 格式未来变化** | 采集器做 schema 版本检查，未知格式告警不崩溃 | 升级采集器解析规则 |
| **数据库损坏** | State Store 可从 Event Store 全量重建（快照只是缓存） | `rebuild_state_store()` 一键恢复 |
| **存储空间不足** | 告警推送祥霭，暂停新事件采集（保留只读） | 清理或扩容后恢复采集 |

> **设计原则**：单点失败不级联。采集、存储、生成、推送四个环节独立，一个环节降级不影响其他环节。

### Phase 2 — 多智能体汇聚

**目标**：第一步接入赫明，两人拼版跑通后接入守山，建立多智能体日报和关系网络

**范围**：
- [ ] 多实例采集架构（赫明 ~/.hermes-genboz/ → collector 部署）
- [ ] 多智能体拼版日报（拼盘式：每人独立叙事 + 可选交汇小节）
- [ ] 关系网络初版（统计 + 趋势曲线，不做 LLM 质量判断）
- [ ] 协作事件检测（接入后先交叉验证数据源，可用则加）
- [ ] 推送策略（每人单推 or 拼版合推，待定）

**接入顺序**：赫明先接入 → 两人拼版跑 1-2 天 → 守山一步接入

**不做**：
- LLM 判断协作质量（无可靠事实锚点）
- 信任度/熟悉度数值（无采集源）
- 力导向图可视化（Phase 3 看板）

**待验证**：群聊 @ 消息跨实例一致性（赫明接入后第一件事）

### Phase 3 — 成长系统 + 看板

**目标**：技能树、自主学习、知识库、看板 UI

**范围**：

- [ ] 自主学习系统（合法性约束 + 学习契约 + 学习路径）
- [ ] 知识库 + 知识图谱
- [ ] 共享看板（祥霭 + 智能体）
- [ ] 祥霭角色系统（主动数据采集 + 技能树 + 知识追踪）

### Phase 4 — 叙事引擎

**目标**：故事线检测和生成

**范围**：
- [ ] 故事线检测器（从事件流中识别叙事弧）
- [ ] 沉默期机制
- [ ] 里程碑系统
- [ ] 多智能体跨视角故事线

## §6 产品边界（明确不做）

```
不做：
├─ 智能体管理面板（那是 Hermes 本身的事）
├─ 效率监控和告警（那是运维系统的事）
├─ 代码级别的项目管理（那是 Git + CI/CD 的事）
├─ 智能体之间的直接通信（那是基础设施层的事）
├─ 任何写入 Hermes 的操作（观察不干预）
└─ 实时对话分析（对齐成本不匹配——等日终批处理）
```

## §7 关键指标（成功标准）

```
用户指标（祥霭）：
├─ 日报打开率（长期保持 > 80%）
├─ 日报阅读完成率（不是扫一眼就关）
└─ 自然互动率（祥霭主动问"展开这个故事"）

智能体指标：
├─ 记忆准确性（LLM 生成内容可追溯到 Event Store 的比例 > 95%）
├─ 故事线质量（祥霭确认"这条故事线有意思"的比例）
└─ 数据完整性（每个智能体的每日事件采集无缺失）

系统指标：
├─ 采集延迟（事件发生到入库 < 30 分钟）
├─ 日报生成延迟（日终批处理 < 5 分钟）
├─ 存储增长（月增量 ~200MB，实测顾远26MB/赫明72MB基础，SQLite完全可控）
└─ LLM 成本（月 < ¥1，实测 DeepSeek-V3 日报约 ¥0.09/月；预留空间给故事线和知识提取）
```

## §8 附录

### 8.1 术语表

| 术语 | 定义 |
|------|------|
| Event Store | 不可变事件日志，追加写入，哈希链签名 |
| State Store | 从 Event Store 计算出的状态快照，可随时销毁重建 |
| 技能化学反应 | 两个技能组合产生的额外效果 |
| 学习契约 | 绑定在学习中技能上的量化考核标准 |
| 知识图谱 | 知识节点之间的语义关系网络 |
| 沉默期 | 故事线生成的最小间隔，防止叙事通货膨胀 |
| 拼版日报 | 同一天所有智能体的视角整合在一份日报中 |

### 8.2 已有数据资产（Hermes → GenArk 可搬运）

| 数据源 | 规模 | → GenArk 维度 |
|--------|------|--------------|

| Memory 系统 | MEMORY.md + USER.md | 智能体知识库 |
