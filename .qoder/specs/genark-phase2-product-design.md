## §1 背景

### 1.1 从哪里来

Phase 1 跑通了顾远单实例的完整数据管道：Event Store → 哈希链 → 日报 → 钉钉推送。Phase 2 要回答一个产品问题：**当有三个智能体时，祥霭每天看到什么？**

### 1.2 守山调研的核心发现

五大框架（CrewAI / AutoGen / MetaGPT / LangGraph / Swarm）**没有任何一个把智能体当成持续成长的独立个体**——它们都视智能体为任务流水线上的临时工人。但它们的可观测性设计有值得借鉴的模式：

| 借鉴来源 | 设计模式 | → GenArk Phase 2 应用 |
|----------|---------|----------------------|
| LangGraph Checkpointer | 每次状态变更自动存检查点 | 事件采集粒度：状态变更 = 一个事件（不只是 tool_use） |
| CrewAI step_callback | 在关键执行节点埋钩子 | 观测点设计：在 session / skill / memory 变化时触发采集 |
| MetaGPT `_observe()` | 按角色过滤关心的消息类型 | 按智能体维度呈现——各自视角，不交叉污染 |
| AutoGen AgentEvent | 标准化事件格式 + OTEL | 事件 schema 标准化（已有基础，Phase 2 扩展 agent 字段） |

### 1.3 创始人决策约束

2026-05-27 范围对齐会议：
- **Phase 2 砍掉 LLM 叙事** → 日报改为纯数据面板（数据质量稳定后再恢复叙事）
- **不做 LLM 协作质量评分**——无可靠事实锚点
- **不做信任度/熟悉度数值**——无采集源
- **不做力导向图可视化**——那是 Phase 3 看板的事

## §2 产品设计原则

在落笔具体设计之前，先确立三条原则——这是区分「监控面板」和「生命叙事」的边界：

```
原则 1 — 呈现成长，不呈现效率
  ❌ "赫明今日完成任务 3 项，代码提交 5 次"
  ✅ "赫明今天修复了一个 P0 漏洞，skill 安全审计 +2"

  评估标准：每一条日报摘要是否回答了"这个智能体今天变成了什么样的人"，
  而非"这个智能体今天干了多少活"。

原则 2 — 各自视角，不揉合成一个故事
  ❌ "今天团队协作完成了 SPY 迭代"
  ✅ 顾远段落："今天产出了 SPY 运营增强的 handoff"
     赫明段落："今天接到 SPY handoff，开始技术设计"
     交汇小节（仅在 @ 证据充足时出现）："两人在 SPY 看板上协作推进"

  评估标准：祥霭能否从日报中读出"三个独立个体的不同一天"。

原则 3 — 数据可追溯，叙事有据可查
  每一条日报摘要必须能追溯到 Event Store 中的具体事件。
  Phase 2 虽砍 LLM 叙事，但纯数据面板同样需要"这个数字从哪来"的可验证性。
```

## §3 观测维度设计

### 3.1 核心问题：哪些事件值得记录？

守山的调研提出了四个观测维度原型：
- 今天学会了什么（新 skill、新 memory、新行为模式）
- 产生了什么判断（做了哪些决策、为什么）
- 和其他智能体有什么互动
- 有什么需要祥霭知道的

Phase 2 将其落地为事件采集的**六大维度**：

```
观测维度一：会话生命体征（Session Vital Signs）
  ├─ 会话次数、时长、消息数
  ├─ 模型消耗（tokens、cost）
  ├─ 工具调用次数与成功率
  └─ 来源：session JSONL 的统计字段（Phase 1 已验证可行）

观测维度二：技能成长（Skill Growth）⭐ 核心差异
  ├─ 技能使用次数变化（从 skills .usage.json 的 use_count diff）
  ├─ 新技能获取事件（skill 文件新增）
  ├─ 技能等级估算（基于使用频率 + 成功率的变化趋势）
  └─ 来源：skills 目录的增量 diff（Phase 1 已验证可行）

观测维度三：记忆演化（Memory Evolution）
  ├─ 记忆条目新增/修改（从 MEMORY.md 的 diff）
  ├─ 记忆主题聚类变化（新增了哪个领域的记忆）
  └─ 来源：memory 文件的增量 diff（Phase 1 已验证可行）

观测维度四：知识获取（Knowledge Acquisition）
  ├─ Phase 2 不做知识图谱，但可记录"今天接触了哪些新概念"
  ├─ 从 session 摘要中提取主题词（规则判定，不用 LLM）
  └─ 来源：session JSONL 的模型字段 + 工具参数中的关键词

观测维度五：协作互动（Collaboration）
  ├─ @ 消息检测（群聊中 @ 其他智能体）
  ├─ 共享文件写入（~/.hermes-team/ 下的文件变更）
  ├─ 看板任务协作（kanban.db 中 assignee 交互）
  └─ 来源：群聊 JSONL 的 @ 模式匹配 + 文件变更时间戳（交叉验证后决定做不做）

观测维度六：决策参与（Decision Participation）⭐ 新维度
  ├─ 方向性对话中的判断输出（从 session 摘要中识别）
  ├─ 文件写入中的"结论"类内容
  └─ 来源：session JSONL 的语义标记（先做简单规则：文件产出 + 长输出 → 标记为潜在决策）
```

### 3.2 观测维度的采集优先级

Phase 2 分两步走，先跑稳基础再扩展：

**第一步（接入赫明后立即跑）**：
- [ ] 维度一：会话生命体征（已有采集逻辑，加 agent 字段即可）
- [ ] 维度二：技能成长（已有采集逻辑，加 agent 字段即可）
- [ ] 维度三：记忆演化（已有采集逻辑，加 agent 字段即可）

**第二步（两人拼版跑通 1-2 天后）**：
- [ ] 维度五：协作互动（依赖交叉验证脚本的结果决定做不做）
- [ ] 维度四 + 六：知识获取 + 决策参与（需先定义规则，不急）

## §4 拼版日报产品形态

### 4.1 一句话描述

祥霭每天 23:00 收到一条钉钉消息，里面是当天三个智能体的**独立数据面板拼盘**——不是一份揉合的报告，而是三个并排的窗口。

### 4.2 日报结构（Phase 2 纯数据版，零 LLM）

└─────────────────────────────────────────────────┘

### 4.3 交汇小节的生成规则

```
✅ 出现条件（同时满足）：
  - 当天有 @ 消息的明确证据
  - 交叉验证通过（两边 JSONL 对 @ 的记录一致）

❌ 不出现的情况：
  - 当天无 @ 互动 → 交汇小节整段省略
  - 交叉验证不一致 → 降级为简单计数（"检测到 3 次互动信号，数据校准中"）
  - 单方数据缺漏 → 标注 *（"数据来自赫明侧"）
```

### 4.4 关系趋势的计算方式

```
输入：近 7 天的 @ 互动次数统计（纯计数，不做 LLM 语义分析）
趋势符号：
  ↗ 本周比上周增加 ≥ 30%
  → 变化 < 30%
  ↘ 本周比上周减少 ≥ 30%
  （首周无基准 → 显示为 —）

不做：
  ❌ 协作质量评分
  ❌ 信任度/熟悉度数值
  ❌ 力导向图
  ❌ LLM 叙事描述
```

## §5 协作事件检测设计

### 5.1 核心问题

两个独立 Hermes 实例之间的交互（@ 消息）是否在各自的 JSONL 中一致记录？这决定了交汇小节的数据可信度。

### 5.2 交叉验证脚本（接入后第一件事）

```
输入：
  - 顾远当日 JSONL
  - 赫明当日 JSONL

验证逻辑：
  1. 从顾远侧提取所有 @ 赫明的消息（时间戳 + 内容摘要）
  2. 从赫明侧提取所有 @ 顾远的消息（时间戳 + 内容摘要）
  3. 交叉比对：顾远发出 @ 的时间戳 ± 5 分钟内，赫明侧应有对应记录

产出：
  - 一致率（如果 > 90% → 数据可用，启用在交汇小节）
  - 不一致样本（供人工判断）
  - 结论：做 or 不做
```

### 5.3 三种结果的处理

| 验证结果 | 处理方式 |
|---------|---------|
| **一致率 > 90%** | 交汇小节启用，数据标注"交叉验证通过" |
| **一致率 50-90%** | 交汇小节启用但降级——只展示双方都确认的互动，标注"部分数据" |
| **一致率 < 50%** | 交汇小节**不做**——改为在日报底部显示一条提示："今日检测到 N 次跨智能体信号，数据一致性不足以展示详情。Phase 3 将改进采集。" |

## §6 推送策略

### 6.1 选项

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| A. 拼版合推 | 三人数据拼成一条消息推送 | 祥霭一次看完全景 | 每人段落受限、可能太长 |
| B. 每人单推 | 三人各推一条独立消息 | 每人详细、独立叙事感强 | 祥霭收到三条消息、缺少"拼图"感 |
| C. 拼版合推 + 关键事件单推 | 日常拼版合推；如果某人当天有重大事件（技能 +5 以上、P0 修复），额外单推一条 | 平衡全景和重点 | 逻辑复杂、需要定义"重大事件"阈值 |

### 6.2 决策

**创始人选择方案 C：日常拼版合推 + 重大事件单推**（2026-06-04 确认）

**日常推送**：每天 23:00，三人数据拼成一条消息推送。
**重大事件单推**：当智能体当天触发以下任一条件时，额外单独推送一条：

```
重大事件阈值（规则判定，不用 LLM）：
  ├─ 技能单日变化 ≥ +5  → "赫明的 security-audit +5 → 66，正在快速成长"
  ├─ 新技能获取          → "顾远解锁了新技能：数据分析"
  ├─ 记忆新增 ≥ 5 条     → "守山今天积累了 5 条新记忆"
  └─ 工具成功率骤降 < 50% → "赫明今日工具成功率骤降至 43%，需关注"
```

**设计要点**：
- 重大事件推送紧跟在拼版日报之后（同一条钉钉消息的第二段，或紧邻的第二条）
- 单推消息风格：一句话叙事（规则模板拼接，不调 LLM）
- 如果当天无重大事件 → 只发拼版日报，无单推
- 如果多人触发 → 按触发顺序依次追加
