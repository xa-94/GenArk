## §2 逐项差距分析

### 2.1 🔴 P0-1: 赫明实例路径过期 → 数据管道断裂

**产品设计要求**（§7）：collector 同时采集顾远 + 赫明

**实际情况**：

```python
# config.py（当前）
AGENT_INSTANCES = {
    "heming": "/home/hermes/.hermes-genboz",  # ❌ 已不存在
}
```

2026-06-02 赫明 profile 迁移至 `~/.hermes/profiles/heming/`，旧路径 `~/.hermes-genboz/` 已删除。后果：

- `collect-heming.sh`（cron 每 30min）：每次 "0 条新事件，扫描 1 个文件"（实际扫空目录）
- `daily-heming.sh`（cron 每天 23:00）：日报显示 "会话 0 | 消息 0 | 工具 0"
- **最后一条赫明事件时间戳**: 2026-06-02（迁移当天），此后数据**完全空白**
- 赫明 daily 仍在推空日报到钉钉（浪费 token + 误导祥霭）

**修复**：改 `config.py` 中 `heming` 路径为 `~/.hermes/profiles/heming/`，一行。

### 2.2 🔴 P0-2: 推送策略未对齐产品设计

**产品设计要求**（§6.2）：方案 C — 日常拼版合推 + 重大事件单推

**实际情况**：

| 组件 | 产品设计 | 实际 |
|------|---------|------|
| 日常推送 | 三人拼版一条消息 | 每人分推（`daily --agent guyuan` + `daily --agent heming` 各推各的） |
| 拼版日报 | 推送到钉钉 | `daily-all` **明确标注"生成不推送"**，只打印到 stdout |
| 重大事件单推 | 规则模板判定阈值 + 紧邻日报告推送 | **零代码** |
| Cron 调度 | 拼版日报替代单人日报 | 仍运行 4 条 cron（两人采 + 两人分推），无拼版推送 |

**差距**：

1. 祥霭每天收到 2 条独立日报，而非 1 条拼版 — 与 §4.1 "祥霭每天收到一条钉钉消息" 矛盾
2. 重大事件单推的四个阈值（技能≥+5、新技能、记忆≥+5、成功率<50%）代码中完全不存在
3. `composer.py` 的 `_assemble()` 生成拼版文本但无推送逻辑
4. `relations.py` 的关系趋势数据未整合进拼版日报

### 2.3 🟡 P1: 交汇小节未做交叉验证门禁

**产品设计要求**（§4.3）：
- 交汇小节仅在交叉验证 >50% 时才出现
- <50% → 整段省略，改为底部提示

**实际情况**：

`composer.py:detect_intersections()` 直接从 events 表查 @ 消息并展示，**不检查交叉验证结果**。当前交叉验证结果是 77.2%（<80% 不可靠），但 `daily-all` 的输出仍会包含交汇小节。

**修复方向**：`detect_intersections()` 调用前先跑一次 mini 交叉验证（当日数据），根据结果决定展示级别。

---

### 2.4 🟡 P1: 关系统计的产品形态差异

**产品设计要求**（§4.2 日报结构）：
```
顾远↔赫明: 近7天协作 8 次 · 趋势 →
顾远↔守山: 近7天协作 3 次 · 趋势 →
赫明↔守山: 近7天协作 5 次 · 趋势 →
```

**实际情况**（`relations.py`）：
- 按 agent 维度统计各自发起的 @ 次数，非 pair-wise
- 有 Unicode 迷你图（`▁▃▅█`）但未接入拼版日报
- `daily-all` 的 `_assemble()` 完全没有关系统计区块

**修复方向**：将 `compute_relations()` 改为 pair-wise 统计，在 `_assemble()` 末尾追加关系趋势区块。
