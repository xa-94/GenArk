### 2.2 核心评估点

**数据采集**
- Hermes 会话 JSONL 的增量采集方案是否可行？
- Memory / Skills 文件的 diff 检测方案？
- 跨实例（顾远/赫明/守山）的数据汇聚方案？

**存储方案**
- Event Store（不可变 + 哈希链）+ State Store（可重建快照）的双层架构
- 推荐的是 SQLite，你评估是否合适？有无更好选择？

**LLM 集成**
- 日报叙事生成、故事线检测、知识提取等对 LLM 的依赖
- 成本估算是否合理？有无风险点？
- LLM 输出可追溯性（引用 event_id）的技术可行性？

**核心模块技术风险**
- 知识图谱的存储和查询（关系类型 IS_A / REFERENCES / CAUSES...）
- 遗忘引擎的衰减公式实现
- 技能化学反应的计算模型
- 共享看板（智能体也能读取）的权限模型

### 2.3 Phase 1 最小可行方案

请重点评估 Phase 1 的范围：
- 只覆盖顾远一个实例
- 只做数据采集 + Event Store + 日报生成
- 不做 UI、不做知识图谱、不做自主学习

Phase 1 是否可以快速跑通？预计工作量？

### 2.4 风险点

- 对 Hermes 系统的改造范围（是否需要新增工具如 `alp_report`？）
- 数据采集对 Hermes 性能的影响
- LLM 幻觉风险及护栏方案的有效性
- 存储增长预估是否合理？
